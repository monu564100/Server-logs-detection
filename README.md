# LogAnomalyPlatform

LogAnomalyPlatform is an end-to-end log anomaly detection system for server security monitoring.
It parses raw logs, engineers time-window features, trains an unsupervised model, predicts attack windows,
exports reports, launches a dashboard, and sends email alerts with Excel attachments.

## 1) Complete Flow Overview

When you run `server.py`, the system executes this sequence:

1. Run pipeline script (`src/pipeline.py`)
2. Parse raw logs to structured format
3. Clean and normalize timestamps/levels/IP/service fields
4. Aggregate data into 5-minute feature windows
5. Train `IsolationForest` and generate anomaly predictions
6. Label attacks + severity + confidence metrics
7. Export anomaly reports to Excel
8. Build `critical_ips.xlsx`
9. Send security email with:
   - `data/anomalies.xlsx`
   - `data/critical_ips.xlsx`
   - recommended server protection actions
10. Launch dashboard (`src/dashboard.py`) on `http://127.0.0.1:8050`

## 2) Data We Are Using

### Primary data source

- Local log file: `data/Linux.log`

### Additional supported sources

- Any log file/folder passed with `--input`
- File patterns searched recursively: `*.log`, `*.txt`, `*.out`
- If no extension-matched logs are found, files without extension are also considered

### Parsed schema

Raw log lines are transformed into these columns:

- `timestamp`
- `service`
- `level`
- `message`
- `ip`
- `source_file`

## 3) Data Cleaning and Preprocessing

The cleaning logic is rule-based and implemented in `src/pipeline.py`.

### 3.1 Parsing methods

- Timestamp extraction via regex (`TS_REGEX`), converted using `pandas.to_datetime`
- Log level extraction via regex (`LEVEL_REGEX`) with normalization:
  - `WARNING` -> `WARN`
- IP extraction via IPv4 regex (`IP_REGEX`)
- Message extraction from text after `:`
- Service extraction from log token structure, with fallback to folder name or `linux`

### 3.2 Cleaning rules

- Empty/blank lines are skipped
- Unreadable files are skipped safely
- Rows with invalid timestamps are repaired or dropped:
  - If source line has no timestamp, a synthetic timestamp is generated to preserve order
  - Rows still invalid after conversion are dropped
- All logs are sorted by timestamp
- Unknown IPs are stored as `Unknown`

### 3.3 Output after cleaning

- Structured logs are saved to:
  - `data/parsed_logs.parquet`

## 4) Feature Engineering Methods

Data is grouped into `window_minutes` bins (default: 5 minutes) using timestamp floor.

### Engineered feature columns

- `log_count`: number of log messages in the window
- `error_ratio`: fraction of `ERROR` logs in the window
- `warn_ratio`: fraction of `WARN` logs in the window
- `avg_message_len`: average log message length
- `unique_services`: distinct services in the window
- `unique_ips`: distinct non-unknown IPs in the window

Feature table output:

- `data/features.parquet`

## 5) Model Used for Training and Prediction

### Model

- `IsolationForest` (`scikit-learn`)
- Mode: unsupervised anomaly detection

### Training pipeline

1. Select numeric feature columns listed above
2. Fill missing values with `0`
3. Scale features using `StandardScaler`
4. Train `IsolationForest` with defaults:
   - `contamination=0.05`
   - `n_estimators=200`
   - `random_state=42`

### Prediction outputs

- `anomaly`: model class (`-1` anomalous, `1` normal)
- `anomaly_score`: decision function value (lower = more anomalous)

### Post-processing labels

- `attack_prediction`:
  - `Attack` if anomaly (`-1`)
  - `Normal` if normal (`1`)

- `severity` for attack windows:
  - Quantile-based split by anomaly score among attack rows
  - Most anomalous third -> `Critical`
  - Next third -> `High`
  - Remaining -> `Medium`

### Confidence and accuracy-level metrics

- `prediction_confidence_pct`:
  - risk-normalized confidence per row (50-100 scale)
- `prediction_accuracy_level_pct`:
  - separation proxy from normal vs attack score distributions
  - bounded to practical range

Training outputs:

- `data/features_with_anomaly.parquet`
- `data/training_report.json`

## 6) Excel Reporting

### Anomaly report (`data/anomalies.xlsx`)

Generated by `export_anomalies_to_excel` with sheets:

- `anomaly_windows`
- `anomaly_logs`

### Critical IP report (`data/critical_ips.xlsx`)

Generated by `server.py` with sheets:

- `critical_ips` (aggregated top IP stats)
- `critical_log_rows` (raw rows from critical/attack windows)
- `critical_windows` (window-level records)

## 7) Dashboard and Monitoring

Dashboard (`src/dashboard.py`) reads parquet outputs and visualizes:

- Volume vs anomaly windows
- Error ratio over time
- Top IPs
- Critical IPs derived from attack windows
- Raw logs for selected anomaly window

Includes:

- Service-level filtering
- Refresh pipeline button
- Excel export button
- Key KPI cards (anomaly %, unique IPs, critical IPs, confidence, accuracy proxy)

## 8) Email Alerting Flow

Email is sent every time `server.py` runs.

### Required `.env` settings

- `EMAIL_USER`
- `EMAIL_PASSWORD`
- `EMAIL_PORT` (typically `587`)

### Optional `.env` settings

- `EMAIL_HOST` (default: `smtp.gmail.com`)
- `EMAIL_TO` (default fallback is configured in code)

Email includes:

- Subject with timestamp
- Security recommendation checklist
- Attachments:
  - `data/anomalies.xlsx`
  - `data/critical_ips.xlsx`

## 10) How To Run

### Install dependencies

```bash
pip install -r requirements.txt
```

### Run full application flow

```bash
python server.py
```

### Run only pipeline (without dashboard/email orchestration)

```bash
python src/pipeline.py
```

### Useful optional arguments for pipeline

```bash
python src/pipeline.py --step full --input data/Linux.log --window-minutes 5 --contamination 0.05 --n-estimators 200 --random-state 42
```

## 12) Notes and Limitations

- This is unsupervised detection: it predicts unusual behavior, not guaranteed attack truth labels.
- Accuracy value is an internal separation proxy, not supervised classification accuracy.
- Severity labels are relative to current run distribution.
